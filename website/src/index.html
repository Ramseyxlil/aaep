An open standard for AI agents to communicate with assistive technology.

Why AAEP exists

When an AI agent acts on your behalf — sends an email, books a flight, changes a record — assistive technology has no standard way to know what's happening. AAEP is that standard.

Real-time, not after the fact

Screen readers, AAC devices, and other AT receive events as the agent is acting — not after a wall of text appears. Users can follow along, intervene before irreversible actions, and stay informed without sighted oversight.

Safety by default

Irreversible high-risk actions (sending email, transferring money, deleting files) emit a confirmation event with default_decision: "reject". If the user doesn't respond, the action doesn't happen. This isn't optional — it's a normative requirement.

Multilingual from day one

AAEP ships v1.0.0 with first-class support for Yoruba, Hausa, and Igbo through the multilingual extension — three of West Africa's most widely-spoken languages. Other accessibility specs added non-English support years after launch. We started there.

Built for the long term

v1.x is supported for at least 5 years with full features, plus 2 years of security-only maintenance. AT software has long product cycles; the protocols it integrates with must match that horizon.

How it works

AAEP defines event semantics. Producers (AI agents) emit events; subscribers (assistive technology) consume them. The transport is up to the implementer — HTTP/SSE, WebSocket, IPC, anything that carries JSON.

Here's what an agent emits when it's about to send an email: 

{
  "type": "aaep:agent.awaiting.confirmation",
  "session_id": "sess_abc123",
  "timestamp": "2026-06-30T14:30:00Z",
  "urgency": "critical",
  "summary_normal": "Confirm sending email to alice@example.com",
  "action": "Send email to alice@example.com with subject: Hello",
  "consequence": "This action cannot be easily undone.",
  "risk_level": "high",
  "irreversible": true,
  "default_decision": "reject",
  "reply_token": "tok_xyz789",
  "timeout_seconds": 300
}

A subscriber (an NVDA add-on, a web React component, a Narrator bridge) receives this event, announces it to the user, waits for their decision, and replies. If the user does nothing, the action is rejected by default. Safety is not opt-in.

Get started

For agent developers

You want your agent to be accessible to AT users.

  1. Read the Quickstart Guide
  2. Pick a reference producer matching your stack (Python, TypeScript, LangChain, Anthropic SDK, Microsoft Agent Framework)
  3. Wire AAEP emission into your existing agent
  4. Verify with the conformance suite

For AT integrators

You build screen readers, AAC software, or other assistive technology.

  1. Read the Subscribers Guide
  2. Study the reference subscriber
  3. Build your AT-specific subscriber by replacing the rendering layer
  4. Run the conformance suite against your subscriber

For procurement and compliance

Who's using AAEP

As of v1.0.0 launch, AAEP is new. The adopters list starts empty by design — we don't pad it with aspirational entries. Real adoption takes time, and we'd rather start honest than start padded.

If your organization is evaluating AAEP, see the listing process. Production-shipping adopters can be the first entry.

What we commit to

  • Backward compatibility within v1.x for at least 5 years
  • Security patches for v1.x through June 2033 (5 years full + 2 years security-only)
  • Open governance with a defined path to broader Steering Committee as adoption grows
  • Open foundation transfer when adoption thresholds are met (5+ implementations across 3+ organizations on 3+ continents)
  • No paid certifications — conformance is objective and verifiable via the suite
  • No vendor lock-in — spec is CC-BY-4.0, code is MIT
